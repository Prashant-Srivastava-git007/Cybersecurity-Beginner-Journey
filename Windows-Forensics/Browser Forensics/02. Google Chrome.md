# üîç Google Chrome Forensics Guide

Google Chrome stores a significant amount of user activity, making it a valuable source of forensic evidence. This guide explains key Chrome artifacts, where they are stored, and how to examine them using tools like **DB Browser for SQLite** and **Hindsight**.

---

## üìÇ Chrome Browser Data Location

Chrome stores its browsing data in:

```
C:\Users\<UserName>\AppData\Local\Google\Chrome\User Data\Default
```

This directory contains the **History** SQLite database, which holds website visit information, downloads, and user navigation patterns.

---

## üìò What Chrome History Contains

Chrome‚Äôs history database helps analysts identify:

* Websites visited in the last X days
* URLs, page titles, and referrer information
* How many times each site was visited
* Exact timestamps of each visit
* The user profile associated with the activity

Tool used for analysis: **DB Browser for SQLite**

To open the database:
**DB Browser ‚Üí Open Database ‚Üí Navigate to the History file in the Default folder**

---

# üóÇÔ∏è Important Tables in Chrome‚Äôs History Database

## 1. **urls Table**

This table stores general information about visited pages, including:

* URL
* Page title
* Visit count
* Last visit time
* Typed count (how often the URL was manually typed)

This helps identify frequently visited sites and user intent.

---

## 2. **visits Table**

Tracks every individual visit, including:

* Timestamp of the visit
* Transition type (how the user reached the URL)
* Referring visit (previous page in the navigation chain)

This table is essential for reconstructing user browsing behavior.

---

# üîÑ Understanding Transition Types

The **transition field** tells *how* a visit occurred. It contains:

* **Core transition type** (last 8 bits)
* **Qualifiers** (additional attributes)

---

## üß© Core Transition Types

| Decimal | Type              | Description                                      |
| ------: | ----------------- | ------------------------------------------------ |
|       0 | LINK              | User clicked a link from another page            |
|       1 | TYPED             | User manually typed the URL                      |
|       2 | AUTO_BOOKMARK     | Visit triggered from bookmarks                   |
|       3 | AUTO_SUBFRAME     | Visit triggered automatically by iframe/subframe |
|       4 | MANUAL_SUBFRAME   | User manually navigated inside a subframe        |
|       5 | GENERATED         | Visit from keyword search (omnibox)              |
|       6 | START_PAGE        | Visit from browser start page/new tab            |
|       7 | FORM_SUBMIT       | Triggered by form submission                     |
|       8 | RELOAD            | Page reload (button or F5)                       |
|       9 | KEYWORD           | Visit from search engine keyword                 |
|      10 | KEYWORD_GENERATED | Keyword-initiated visit                          |

# üßÆ Useful SQLite Queries

### Extract Core Transition Type

```sql
SELECT id, visit_time, transition,
       transition & 0xFF AS core_transition
FROM visits;
```

### Convert Visit Duration to Readable Format

```sql
SELECT id, url, visit_time, visit_duration,
       visit_duration / 1000000 AS duration_seconds,
       (visit_duration / 1000000) / 60 AS duration_minutes,
       (visit_duration / 1000000) % 60 AS duration_mod_seconds,
       printf('%d:%02d',
              (visit_duration / 1000000) / 60,
              (visit_duration / 1000000) % 60) AS formatted_duration
FROM visits;
```

---

# ‚è±Ô∏è Visit Duration Explained

The `visit_duration` field measures how long (in microseconds) a user stayed on a webpage.
This helps determine **engagement**, **interest level**, or **suspiciously short visits**.

---

# üîó Correlating URLs and Visits

For a full understanding of browsing behavior:

* **urls table** ‚Üí where the user went
* **visits table** ‚Üí how and when they went there

Cross-referencing these tables gives a complete picture of navigation paths and user intent.

# üõ†Ô∏è Chrome Analysis Using Hindsight

**Hindsight** is a browser forensics tool that extracts and parses Chrome (and Edge) history.

### What Hindsight Can Do

* Extract browsing history, cookies, downloads, cache
* Parse SQLite and local storage files
* Generate human-readable reports
* Create timeline views
* Filter and keyword search
* Export data for further analysis
* Supports Chrome + Edge Chromium

---

## ‚ñ∂Ô∏è Running Hindsight

Example command:

```
hindsight.exe -i "C:\Users\<UserName>\Desktop\Kape_Out_browser\C\Users\<UserName>\AppData\Local\Google\Chrome\User Data" -o C:\Users\<UserName>\Desktop\Case_Files\WebBrowser\Chrome
```

### Breakdown:

* **hindsight.exe** ‚Üí The tool
* **-i** ‚Üí Input directory containing Chrome data
* **-o** ‚Üí Output directory where the report will be saved

After running the command, Hindsight displays a processing summary.

---

# üìä Viewing Hindsight Output

1. Open **EZViewer** (found in the EZTools folder).
2. Load the `.xlsx` report generated by Hindsight.
3. Use filters to simplify the view:

   * Filter by **Type** such as `url (visited)` or `downloads`.
   * View timestamps, downloaded files, visit durations, transitions, and more.

This allows quick identification of browsing behavior, suspicious activity, and timeline-based patterns.
